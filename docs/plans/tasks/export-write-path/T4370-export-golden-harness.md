## Problem

The export pipeline is about to be consolidated (5 finalize copies → 1, 3 final_videos writers → 1, pipelines out of routers, FFmpeg/interpolation dedup). None of that is safe without characterization tests pinning CURRENT behavior — the refactors must prove output parity, not argue it.

## Solution

Two harness layers:

1. **DB-effects characterization.** For each export trigger (single-clip render, multi-clip Modal branch, multi-clip local branch, overlay final, durable worker, sweep auto-export): a fixture project → run the trigger with rendering stubbed at the FFmpeg/Modal boundary → snapshot-assert the complete DB delta (working_videos/final_videos/export_jobs/projects rows: every column). These become the parity oracle for T4380-T4410. Use the local (`MODAL_ENABLED=false`) path from T4120; stub at the subprocess boundary so tests are fast and deterministic.
2. **Render goldens (small, real).** 2-3 tiny fixture videos (seconds long, generated by ffmpeg in-test or checked in ≤ 1MB) through the LOCAL render path with fixed keyframes/segments: assert output properties (duration ±1 frame, resolution, stream layout via ffprobe) and perceptual similarity on sampled frames (e.g., compare frame hashes with tolerance — pixel-exact will flake across ffmpeg builds; document the tolerance). These become the oracle for T4430 (FFmpeg params) and T4420 (interpolation): local-vs-Modal parity is asserted on the SAME interpolation fixtures at the unit level (given keyframes + frame index → identical crop rect from both implementations).
