## Goal

A parent annotates **team** plays alongside their own athlete's plays, then drops ONE link into
the team WhatsApp/TeamSnap — *"here's the game"* — and every family on the team can:

1. **Watch instantly** — the link unfurls with a poster and plays the **team recap** with no
   account and no signup wall (~edge-page latency, T4840 class).
2. **Claim the game** — "Add this game to your account" → sign up → choose whether to import
   the team annotations along with the game.
3. **Engage deeper** — tag *their* athlete on the imported game, build reels, share back.

Every shared game is an acquisition channel into the exact audience we want: the other ~15
families on the same team, watching content they already care about. This is the app's first
**watch-first** growth loop (the Dual-Camera epic's invite links are join-first).

## Locked Product Decisions (user, 2026-07-21 — reference these, don't re-ask)

1. **Strictly one layer per clip.** Every annotation is either **My Athlete** or **Team**,
   never both. The "my kid scored a team-worthy goal" case is handled by player tags on Team
   plays (decision 7), not dual membership.
2. **The layer IS `raw_clips.my_athlete` — no new schema, no migration.**
   `my_athlete = 1` (or NULL, pre-migration legacy) = My Athlete layer;
   `my_athlete = 0` = Team layer. Materialized shared-in clips are already forced
   `my_athlete = 0` + `shared_by` NOT NULL — they land on the recipient's Team layer with
   provenance intact. All downstream consumption already respects the split
   (`exclude_teammate_reels_clause` keeps reels/rankings/collections on `my_athlete=1`).
3. **Anonymous visitors watch the team recap only.** The full game (and the athlete-layer
   content) is what claiming gets you. No signup wall before the recap plays.
4. **Layer toggle is sticky within the session.** New clips land on the currently-selected
   layer; the toggle resets to My Athlete each time a game is opened. Ephemeral view state —
   NEVER persisted (no-persisted-view-state rule).
5. **The combined all-clips recap is replaced** by two per-layer recaps: **Team Recap** and
   **{Athlete} Recap**. No third "everything" view.
6. **Links live until revoked** (public-reel-link semantics, idempotent per game, revocable
   from the game card). After the game source expires, the link degrades to recap-only
   playback (recap videos survive expiry) and claiming imports annotations without the
   full-game source.
7. **Player tags on Team plays** (`tagged_teammates`, already exists and already travels
   through materialization) are the discovery mechanism: each importing family filters the
   team layer to *their* kid's moments.
8. **Import is a consent moment.** The claim dialog offers: game + team annotations, or game
   only. (Game always; annotations opt-in.) Multi-profile accounts pick the athlete profile
   here.

## Architecture Decisions (reference from task files, don't duplicate)

1. **Reuse, don't parallel-build.** Token pattern from `shares.py`; claim materialization via
   `materialize_game_share` (`app/services/materialization.py`) — game travels as R2
   references (`_copy_game` + `_find_existing_game_by_hashes` dedup + `_create_storage_refs`);
   the game source is stored ONCE globally (`games/{blake3}.mp4`, content-addressed) so claims
   add no storage cost.
2. **T5330 NUF-blindness is an invariant.** Every claim path routes through
   `materialize_game_share`/`_copy_game`/`_materialize_clips` so copied games/clips get a
   non-null `shared_by` (precedence `sharer_email -> sharer_user_id -> "lost"`). A claim path
   that leaves `shared_by` NULL regresses T5330 (onboarding silently skipped). The recipient's
   OWN athlete tagging afterwards is genuine content and counts for quests.
3. **Edge page pattern = T4840/T4890.** New two-segment edge function
   `functions/shared/game/[token].js` (the existing single-segment `/shared/{token}` matcher
   does not collide); OG unfurl uses the recap poster pipeline (T5180/T5270: warm at share
   creation, stable proxy URL in og:image, SPA fallthrough on any doubt).
4. **Stitch-on-share.** Recaps are generated by auto-export today (often at expiry). The
   share-creation gesture ensures the TEAM recap is stitched + poster warmed before the link
   exists (T5270 precedent: the link can't be pasted before creation returns).
5. **Coordinate token/claim plumbing with Dual-Camera T5500** — whichever lands first owns the
   token-landing-claim plumbing; the other reuses it (recorded in both epics).
6. **Existing email teammate share stays untouched.** It serves a different job (targeted
   per-player clip delivery with per-tag filtering). This epic adds the broadcast link path.

## The layer model (summary)

| DB state | Layer | Visible as |
|---|---|---|
| `my_athlete = 1` or NULL | My Athlete | cyan markers/chips; feeds reels, rankings, collections, {Athlete} Recap |
| `my_athlete = 0`, `shared_by` NULL | Team (authored by me) | amber markers/chips; feeds Team Recap + the shared link |
| `shared_by` NOT NULL | Team (imported) | amber + "Shared by {name}" attribution; quest-blind |

- [ ] Annotate has a My Athlete / Team layer model: mode toggle for new clips, per-clip layer
      switch (desktop + mobile), color-coded markers/chips/filters

      layer (regression-tested, no behavior change)
- [ ] Recap experience is split: Team Recap + {Athlete} Recap (stitched + posters per layer);
      the combined recap is gone

      imported content (T5330 regression-tested)
- [ ] Revocation works; expired-source games degrade to recap-only links
