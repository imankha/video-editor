# T1160 + T1170: DB Hygiene — Row Pruning + Size-Gated VACUUM

**Status:** DESIGN (awaiting approval)
**Branch:** `feature/T1160-T1170-db-hygiene`
**Bundled because:** T1160 creates the freed space that T1170's VACUUM reclaims. Running them separately would either VACUUM nothing (T1170 first) or leave freed pages unreclaimed (T1160 first).

---

## Current behavior

### Where cleanup runs today

[session_init.py:129-133](src/backend/app/session_init.py#L129-L133) calls `cleanup_database_bloat()` once per user per server process lifetime (gated by `_init_cache`). This is the user's first request — a genuine gesture, not reactive.

Before that, the same init block calls `cleanup_stale_restored_projects()` (re-archives projects idle > N days).

### What `cleanup_database_bloat()` does today

[project_archive.py:341-397](src/backend/app/services/project_archive.py#L341-L397):

1. `DELETE FROM working_videos` where not the latest version per project.
2. `DELETE FROM export_jobs` where status in (complete, error) and `completed_at < now-7d`.
3. If either delete touched any rows: commit, then unconditional `VACUUM`.

### Gaps

- **`working_clips` old versions** never pruned. Each re-export creates a row; only latest is read. ~300-500 bytes × dead versions × N clips.
- **`before_after_tracks`** rows never pruned. Each overlay export writes new rows tied to a `final_video`; only the project's current `final_video_id` is read.
- **`modal_tasks`** terminal rows never pruned.
- **VACUUM is unconditional** — wasted I/O on small DBs; no re-trigger once server is long-running.

## Target behavior

### Pruning rules (additive to `cleanup_database_bloat`)

All three run inside the existing `get_db_connection()` context, before the (now gated) VACUUM. Each logs its count via the existing result dict.

#### Rule A — working_clips: keep only latest version per identity

Identity = `COALESCE(rc.end_time, wc.uploaded_filename)` — matches [queries.py:63 `latest_working_clips_subquery`](src/backend/app/queries.py#L63) which every read-path uses. Verified (see T1160-T1170 conversation log): all reads either already filter via this subquery, do primary-key lookups on ids returned by filtered lists, or benefit from pruning (current full-project scans emit duplicates).

```python
from app.queries import latest_working_clips_subquery

cursor.execute(f"""
    DELETE FROM working_clips
    WHERE id NOT IN ({latest_working_clips_subquery(project_filter=False)})
""")
result["working_clips_pruned"] = cursor.rowcount
```

`project_filter=False` → scans all projects, no placeholder. Reuses the production subquery; if "latest" semantics ever change, one place to update.

#### Rule B — before_after_tracks: keep only tracks for each project's current final_video

```sql
DELETE FROM before_after_tracks
WHERE final_video_id NOT IN (
    SELECT final_video_id FROM projects WHERE final_video_id IS NOT NULL
)
```

Note: FK is `ON DELETE CASCADE` from `final_videos.id`. Today nothing prunes `final_videos` rows, so tracks for old final_videos sit indefinitely. This DELETE reaches them directly without waiting on `final_videos` pruning.

#### Rule C — modal_tasks: prune terminal rows older than 24h

```sql
DELETE FROM modal_tasks
WHERE status IN ('complete', 'error', 'cancelled')
  AND COALESCE(completed_at, created_at) < datetime('now', '-1 day')
```

Uses `completed_at` when present, falls back to `created_at` for rows that never recorded a completion timestamp.

### VACUUM gate

Replace the unconditional VACUUM block at the end of `cleanup_database_bloat()`:

```python
from app.database import DB_SIZE_WARNING_THRESHOLD  # 400 * 1024

# ... after commit ...
db_path = _profile_db_path_for_current_user()  # existing helper or inline
size_before = db_path.stat().st_size
if size_before > DB_SIZE_WARNING_THRESHOLD:
    logger.info(f"[Cleanup] DB {size_before // 1024}KB > {DB_SIZE_WARNING_THRESHOLD // 1024}KB — VACUUM")
    conn.execute("VACUUM")  # must be outside any transaction; commit happened above
    size_after = db_path.stat().st_size
    logger.info(f"[Cleanup] VACUUM: {size_before // 1024}KB -> {size_after // 1024}KB")
else:
    logger.debug(f"[Cleanup] DB {size_before // 1024}KB under threshold — skipping VACUUM")
```

Threshold `DB_SIZE_WARNING_THRESHOLD` already defined at [database.py:54](src/backend/app/database.py#L54). Reusing it keeps the two size-related knobs (warn / VACUUM) aligned.

### Lifecycle (unchanged)

Runs from `user_session_init` → `cleanup_database_bloat()`, once per user per server process. No new call sites, no reactive triggers.

## Risks

### FK coverage — what else references the rows we're deleting?

- **working_clips**: referenced by … grep showed no FKs pointing *to* `working_clips.id`. Other tables hold `project_id` / `raw_clip_id` but not `working_clip_id`. Safe.
- **before_after_tracks**: no inbound FKs. Safe.
- **modal_tasks**: no inbound FKs. Safe.

(If a future migration adds an FK to any of these, the DELETE will be blocked by SQLite FK enforcement, surfacing the dependency — better than silent orphaning.)

### VACUUM concurrency

VACUUM requires an exclusive lock and **cannot run inside a transaction**. Current code already calls `conn.execute("VACUUM")` after `conn.commit()`, so precedent holds. `get_db_connection()` uses per-request connections (no shared pool); at session-init time the user has one in-flight request. Acceptable.

If another user's request arrives mid-VACUUM, SQLite returns SQLITE_BUSY. The backend already retries on busy (verify via existing test pattern). No new hazard introduced by *gating* the VACUUM; same behavior as today, just less frequent.

### Request-in-flight reading a row we're about to prune

Session-init runs on the user's first request *before* returning. Subsequent requests from the same user see the post-cleanup state. A concurrent request from the same user at the same moment could theoretically race, but (a) session_init is serialized by the init cache check, (b) pruned rows are non-latest and no endpoint returns their ids to the frontend, so no cached id can target them.

### R2 sync + VACUUM

VACUUM rewrites the file in-place (creates a temp journal, swaps). The file size changes → next write triggers an R2 upload at the new size. The version-tracking in `_user_db_versions` is driven by writes, not file mtime, so version still increments correctly on subsequent mutations. No change needed.

### Pruning rules pruning "too much"

Each rule is self-contained and reversible by simply not running on the next init. The worst case is losing replay-debug context (an errored modal task older than 24h). Acceptable given the 24h window.
